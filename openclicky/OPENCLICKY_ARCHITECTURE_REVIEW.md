# OpenClicky Architecture Review — Computer Use, Provider Routing, Agents, Prompts, Voice, Learning Loop

Lead reviewer's final pass. All high/medium verification corrections applied. Code claims cited as `file:line`. This is a review — no edits made.

---

## 1. Executive Summary

1. **The two computer-use backends are inverted in quality vs. wiring.** The background runtime (BCU) has the better actuation primitive (`SLPSPostEventRecordTo` + `SLEventPostToPid`: no cursor warp, no app activation) but its click/scroll/drag routes are completely unwired. The native Swift path has the inferior primitive (`CGEvent` at `.cghidEventTap`, which warps the real cursor — `OpenClickyComputerUseRuntime.swift:1110-1144`) yet is the only live click dispatcher. There is **no `clickUsingSelectedComputerUse`** — `clickUsingNativeComputerUse` is unconditional (`CompanionManager.swift:6364`), so the "Background Computer Use" toggle is dishonest for clicks. Fixing the selector is the single highest-leverage CUA change.

2. **The most accurate pointing reasoner is wired only to a decorative overlay, with a narrow exception.** The `computer_20251124` beta tool (`ElementLocationDetector.swift`) is the most accurate single-shot pointer per D1, but its only call site is proactive cursor-overlay pointing (`CompanionManager.swift:15799`). It feeds a *real* click only through the cached `prefersLastPointedElement` path (within 120s, `CompanionManager.swift:6389-6402`); the primary/uncached click path uses a weaker `[POINT:x,y]` standard-vision tag. Routing uncached clicks through the beta detector is a contained accuracy win.

3. **The realtime model already makes the IT-vs-agent decision and OpenClicky throws it away.** OpenAI realtime sessions carry `openclicky_use_computer` / `openclicky_start_background_agent` routing tools (`ElevenLabsTTSClient.swift:2062-2095`). When the model fires one, `routeRealtimeToolCall` logs the tool name then discards it and re-runs the raw transcript through the 100%-regex cascade (`CompanionManager.swift:4075`). The most capable provider's own classification is overridden by the weakest classifier. This is the cheapest high-value fix in the codebase.

4. **The one genuine voice capability gap is inline web search.** Every factual question ("weather", "latest price") is shipped to a full Codex background agent because the voice-response inference attaches no `web_search` tool and the Claude Agent SDK bridge is tool-less (R5). D1 (Claude server-side web_search) and D3 (Codex first-party search) confirm this is fillable, not a model limit. Everything else (app launch, volume, type, key, click, Spotify, reminders, messages search) already works locally.

5. **Agent Mode is Codex-only, and the catalog's provider tags do not change that.** `CodexAgentSession` always spawns the bundled Codex CLI (`CodexAgentSession.swift:736-742`); there is no provider branch (`CompanionManager.swift:3098-3111`). No UI path can select an Anthropic model for Agent Mode — all three pickers bind to `codexActionsModels` (gpt-5.x only), and `codexActionsModel(withID:)` coerces any non-gpt-5.x ID to gpt-5.5 (`OpenClickyModelCatalog.swift:149-151`). There is **no Claude agentic Agent-Mode path at all.** The `.openAI` tag on `codexActionsModels` is inert metadata.

6. **The money rule is honored where it bills.** After verification, no genuine mis-billing violation survives. The `ElementLocationDetector` direct REST is **not** a violation — the `computer_20251124` beta tool has no Agent SDK surface (D1), so direct HTTP is the only available transport. `fastFriendlyTitle` tries on-device FoundationModels first (live on this macOS 26 host) and only falls to a 64-token-capped REST call. The one real defect is an **observability bug**: `voiceResponseExecutionFields` reports `api_key_primary` while the runtime is SDK-first (`CompanionManager.swift:3520-3539`) — it mis-reports, it does not mis-bill.

7. **The learning loop is wiring, not a rebuild.** All the pieces exist (rich JSONL logging, review-comment bridge, learn-from-logs / optimize-skills bundled skills, specialist-builder agent, a 30s automation scheduler). What is missing is orchestration: route-miss logging, a scheduled digest, and a dev-mode gate. The previously-claimed repo-operator "blocker" is **not a blocker** — the alias map at `OpenClickyAgentDefinition.swift:112-121` resolves `clicky-repo-operator` to a real on-disk skill on the first candidate. Removing that false blocker strengthens the "wiring not rebuild" conclusion.

8. **`CompanionManager.swift` at 17,194 lines is the dominant structural liability.** ~16,300 of those lines are a single `ObservableObject` class with 1,905 declarations, holding the entire voice routing cascade, provider dispatch, CUA dispatch, and agent launch inline and physically inseparable. Extracting the routing cascade into a `VoiceRouter` is the precondition for nearly every other cleanup.

## 2. Sub-Question Analysis

### 2.1 Computer Use & Control — Native CUA vs Background Runtime

**VERDICT: Yes, they can and should work together — but today they don't, because the selector is dishonest for clicks and the backend with the better primitive (BCU) has its click transport entirely unwired. Treat transport and reasoner as orthogonal axes, make the selector honest first, then compose behind one action protocol.**

**The two orthogonal axes.** Native-vs-background is a *transport* decision (how a click/keystroke actuates). Anthropic-vs-OpenAI-vs-Codex is a *reasoner* decision (who picks the pixel/element). They are independent; conflating them drives most of the catalog confusion (see §4).

- **Native transport:** `CGEvent` posted at `.cghidEventTap` warps the physical cursor and may activate the target app (`OpenClickyComputerUseRuntime.swift:1110-1144`). Keyboard/text can be pid-directed via SkyLight `SLEventPostToPid` or `CGEvent.postToPid` (`Runtime.swift:91-101, 1147-1213`) — keys are background-safe, but clicks are not. There is no pid-targeted mouse path in CoreGraphics.
- **Background transport (BCU):** a separately-bundled app (`xyz.dubdub.backgroundcomputeruse`) exposing an HTTP action server. Its click transport uses `SLPSPostEventRecordTo` for target-only focus then `SLEventPostToPid` for the mouse event — no foregrounding, no cursor warp. **This is BCU's entire reason to exist, and it is dormant.** The controller wires only `pressKey` (`Runtime.swift:343-374`), `typeText` (386-416), `captureFrontmostWindowAsJPEG` (310-341), and discovery helpers (424-467). There is no `click()`, `scroll()`, or `drag()` even though the shipped binary exposes `/v1/click`, `/v1/scroll`, `/v1/drag`, `/v1/set_value`, `/v1/resize` (string-grounded in the binary; verify the route before calling).

**Why the selector is dishonest for clicks.** The dispatch layer has a clean switch for type/key/search (`typeTextUsingSelectedComputerUse` 6346, `pressKeyUsingSelectedComputerUse` 6355, `searchMessagesUsingSelectedComputerUse` 6337). But there is **no `clickUsingSelectedComputerUse`** — `clickUsingNativeComputerUse` (6364) is unconditional, always warping the cursor. The backend enum has exactly two cases (`OpenClickyComputerUseModels.swift:4-6`) and no click branch. Worse, the external-control bridge's click (`clickExternalControlPoint`, `CompanionManager.swift:2504-2549`) calls `nativeComputerUseController.click(at:)` directly, ignoring `selectedComputerUseBackend`. So a user who selects "Background Computer Use" still gets a cursor-warping native click for both voice-routed and bridge-driven clicks. The bundled cua-driver `SKILL.md` explicitly forbids exactly this `cghidEventTap`-to-coordinate pattern.

**Reasoner routing and the beta-tool question.** Two pointing sub-paths route differently:
1. *Click execution* (`clickUsingNativeComputerUse` → `analyzeComputerUsePointingResponse` → `computerUsePointingResolver`, `CompanionManager.swift:15750`): a standard vision call returning a `[POINT:x,y]` tag. None of these use the `computer_20251124` beta tool.
2. *Proactive overlay pointing* (`attemptProactiveElementPointingIfUseful`, 15784): for `.anthropic`, instantiates `ElementLocationDetector` which calls `api.anthropic.com/v1/messages` directly with `anthropic-beta: computer-use-2025-11-24` and the `computer_20251124` tool. This is the **only** use of the specialized pixel-counting beta tool.

**Correction (verified):** the earlier "never fires a real click" framing is refuted. The proactive path calls `rememberPointedElement` (`CompanionManager.swift:15833`), writing `lastPointedElementScreenLocation`; the real click path short-circuits at `6389-6402` and clicks that cached coordinate when `prefersLastPointedElement` is set and the pointed element is <120s old. So the beta detector **can** drive a real click — but only when a proactive pointing occurred in the prior 120s. The uncached/primary click path still uses the weaker `[POINT:x,y]` tag (`parsePointingCoordinates`, 6421/6428).

**Correction (verified):** the `ElementLocationDetector` direct REST is **not** a money-rule violation. Per D1, computer/desktop control is not an Agent SDK built-in tool (SDK tools are Read/Write/Edit/Bash/Glob/Grep/WebSearch/WebFetch; computer-use is a separate API beta or MCP). There is no SDK-equivalent to short-circuit from — direct REST is the only transport. Real per-use cost, but the money rule does not apply. The correct mitigation is gating *when* proactive pointing fires, not forcing it through an SDK that cannot carry the beta header.

The Codex pointing path (`CodexPointDetector`) is Codex acting as a vision reasoner emitting a `[POINT:x,y]` tag — not a true computer-use tool loop (D3: desktop GUI computer-use is a Codex *App* feature, not on the CLI/app-server surface). The subprocess also runs with `--dangerously-bypass-approvals-and-sandbox` / `--sandbox danger-full-access` (`CodexPointDetector.swift:213-216`) — full filesystem access for a task that only returns a coordinate, a screenshot-prompt-injection surface.

**Maturity.**
- *Native foreground CUA:* mature at lower layers (window enumeration, keyboard virtual-keycode mapping, SkyLight pid-directed key delivery, coordinate math), experimental at the pointing-routing layer. Real model-layer test coverage (`OpenClickyComputerUseTests.swift`), but coordinate math (`globalPoint`), `parsePointingCoordinates` edge cases, and the SkyLight fallback are untested. Brittleness: hardcoded SkyLight byte offsets `[24,32,16]` (`Runtime.swift:1207`) unvalidated against macOS 26; auto-enable bypasses the user's disabled setting (`setEnabled(true)` before each action, 6385-6387).
- *Background runtime:* partially integrated and operationally immature. Its differentiating click transport is unwired. Process lifecycle is unmanaged: `startRuntime()` spawns the app via `/usr/bin/open` and forgets the PID (`Runtime.swift:216-274`); teardown omits BCU (`CompanionManager.swift:2595-2596`); `ensureRuntimeReady` trusts a manifest without a liveness probe (277-308), so a stale manifest makes a dead BCU read as ready. Hardcoded dev path `/Users/jkneen/Documents/GitHub/background-computer-use` (`Runtime.swift:138-141`).

Both share a strong architecture: pure Sendable/Codable contracts in `Models.swift`, I/O in `Runtime.swift`, orchestration in `CompanionManager` — model tests stay cheap.

**Best provider/model per backend (grounded in D1/D2/D3).**
- *Native transport (pixel/screenshot pointing):* coordinate precision is load-bearing. Best: **Claude Opus 4.8** (D1: `computer_20251124` with zoom; the pixel-counting training is more accurate than standard vision). Cost/latency alternative: Claude Sonnet 4.6 at medium effort (D1). OpenAI equivalent: gpt-5.5 / gpt-5.4 via the Responses API computer-use tool (D2). Weakest Anthropic path: Haiku 4.5 (older `computer-use-2025-01-24` tool only, D1). **Correction:** do *not* claim "1:1 / no scaling math" for what the app runs — the catalog Opus is the legacy `claude-opus-4-6` (`OpenClickyModelCatalog.swift:88`) and `ElementLocationDetector` defaults to `claude-sonnet-4-6` and downsamples to ≤1366px with explicit coordinate scaling (`ElementLocationDetector.swift:33-37, 106-121`). The 1:1/2576px benefit is Opus-4.8/4.7-only per D1 and is not realized on the current path.
- *Background transport (AX-tree + screenshot):* BCU returns an AX node tree (`/v1/get_window_state`, maxNodes 6500), so element-handle grounding reduces reliance on pixel-perfect vision; a strong reasoner (Opus 4.8 / gpt-5.5) consuming that tree is the fit. Today the tree is fetched but used only for window targeting — its grounding potential is untapped.
- *Catalog finding:* the "Claude Opus" computer-use option points at `claude-opus-4-6`, which D1 confirms is legacy (superseded by 4.7, flagship 4.8). The precision path runs on an outdated model.
- *Realtime models* (gpt-realtime-2 etc.) as computer-use models silently no-op in proactive pointing (`CompanionManager.swift:15817`, the `.openAI`/`.deepgram` case returns with no log). D2/D3 confirm realtime models have no built-in desktop control; they should be excluded from `computerUseModels`.

**Can they work together — the composition.** Yes. The shared plumbing half-exists: BCU's `resolveTargetWindow` already cross-calls the native `frontmostTargetWindow()` (`Runtime.swift:424-430`). The composition is a policy decision over a shared action vocabulary:
- Define one `ComputerUseActuator` protocol (click/type/key/scroll/capture), conformed by both controllers.
- Make the selector honest: add `clickUsingSelectedComputerUse`; implement `controller.click()` against `/v1/click`; route the bridge click through the selector.
- Policy-route by situation, not a global toggle: off-screen / headless / no-cursor-steal / non-frontmost window → background transport; on-screen precision → native (or background once the click transport is proven, eliminating the warp entirely). The reasoner feeds either transport unchanged.
- Consolidate the three duplicate screenshot-pixel→display-point implementations (`CompanionManager.swift:15168`, `ElementLocationDetector.swift:106-121`, `CodexPointDetector.swift:132-137`) into one shared static.

### 2.2 Provider/Model Fit Per Lane

**VERDICT: Each lane has a clear best-fit model, but the catalog ships outdated defaults on the two precision-sensitive lanes (computer-use Opus = legacy 4-6; pointing detector = sonnet-4-6) while the fast-voice and coding lanes are already on-family.**

- **Fast voice (speech-to-speech).** `gpt-realtime-2` is the strongest realtime fit: D2 confirms reliable + parallel tool calling, which is the basis for model-authoritative routing (see §2.5/§2.6). `deepgram-voice-agent` is a viable alternative but lacks a wired tool surface (R4). No model-capability claim here is unverified — both rest on D2/D3.
- **Computer-use (pointing/control).** Best: **Claude Opus 4.8** for precision (D1), gpt-5.5 via Responses API as the OpenAI equivalent (D2). The app ships the legacy `claude-opus-4-6` (`OpenClickyModelCatalog.swift:88`) and the detector defaults to `claude-sonnet-4-6` (`ElementLocationDetector.swift:39`). The grounded improvement is "bump to a current model + the pixel-counting beta tool is more accurate than standard vision" — *not* the unverified-for-this-path "1:1 coordinate" framing, which only holds for Opus 4.8/4.7 and is defeated by the detector's ≤1366px downsample + coordinate scaling.
- **Background coding (Agent Mode).** Anchored on the bundled Codex CLI / app-server (D3: computer-use + background exec + ChatGPT-subscription sign-in, which is the money-rule basis). The selectable models are gpt-5.x (`codexActionsModels`). gpt-5.5 is the strongest coding fit per D2. Note this lane is Codex-only regardless of the provider tag (see §2.3, §4) — Claude is *not* an option for Agent Mode today despite D1 confirming the Claude Agent SDK can run a real agentic loop.

The hardcoded compatibility fallback `gpt-5.4-mini` (`CodexAgentSession.swift:395`, `CodexPointDetector.swift:13`) is embedded in code rather than resolved via the catalog — a silent-failure risk if that ID is renamed (model currency is **unverified** against the bundled Codex runtime version).

### 2.3 Cross-Provider Agent-Handling Consistency

**VERDICT: Consistent at the bottom by accident, fragmented everywhere above it. Every lane funnels into one Codex-only executor — but that unity is a monoculture, not an abstraction; classification, voice-response backends, and the three agent runtimes share no protocol.**

**Consistent convergence.** All agent launches funnel into one path regardless of provider: push-to-talk (`handleFinalVoiceTranscript`), realtime (`routeCompletedRealtimeVoiceTranscriptIfNeeded` 4238), HUD, and programmatic all reach `routeFinalVoiceTranscriptActionIfNeeded` (4392-4440); every agent-positive branch calls `startVoiceAgentTaskPlan` → `startVoiceAgentTask` (11993) → `CodexAgentSession`, which unconditionally launches the bundled Codex CLI (`CodexAgentSession.swift:736-742`). No provider branch at `createAndSelectNewCodexAgentSession` (3098-3111). Consistency-by-monoculture.

**Divergence 1 — classification ignores native provider classification.** Routing is 100% hardcoded regex across ~13 ordered guards (4400-4439); no LLM intent classifier. The realtime session is configured with `openclicky_use_computer` / `openclicky_start_background_agent` (`ElevenLabsTTSClient.swift:2062-2095`); per D2, gpt-realtime-2 has reliable + parallel tool calling. When the model fires a tool, `routeRealtimeToolCall` (4060-4076) logs `toolName` then **discards** it and re-runs the raw transcript through the regex cascade (`CompanionManager.swift:4075`). Deepgram has no tool surface (R4) so only regex routes it.

**Divergence 2 — voice-response backends share no protocol.** `analyzeVoiceResponse` (15268-15321) is a four-arm switch: `.anthropic` → `analyzeClaudeResponse` (SDK-first); `.openAI` → `analyzeOpenAIOrCodexVoiceResponse` (key-first, Codex-fallback); `.codex` → `analyzeCodexVoiceResponse`; `.deepgram` → throws (15306-15311). Adding a fifth provider touches 6+ switch sites. Realtime visual fallback always resolves to gpt-5.4/.openAI (14441-14444) even with only an Anthropic key. `assistantPrefill` is silently dropped on the SDK path (15346 vs `ClaudeAPI.swift:136`).

**Divergence 3 — three background-agent runtimes, no shared protocol.** `CodexAgentSession` (Codex CLI, full agentic loop) is the only runtime wired to Agent Mode. `ClaudeAgentSDKAPI` is a *stateless single-turn query bridge* (delta/result/error only, no tool dispatch) used only for voice responses, never agent tasks. `OpenClickyAgentManager` (Unix-socket daemon) is fully built but **orphaned** — its sole reference is a `@StateObject` in the settings view (`OpenClickyAgentRuntimeView.swift:7`).

**Correction (verified):** the earlier claim "selecting an Anthropic model for Agent Mode gives Codex CLI running an Anthropic model ID" is **false in mechanism**. No UI path lets a user select an Anthropic model for Agent Mode — all three pickers (`CodexAgentModePanelSection.swift:594-596`, `OpenClickySettingsWindowManager.swift:1055-1057`, `ChatHeaderBar.swift:182`) bind to `codexActionsModels`, which contains only gpt-5.x IDs. `setModel` coerces input through `codexActionsModel(withID:)` (`CodexAgentSession.swift:567-572`) → `codexActionsModels.first{...} ?? codexActionsModels[0]` (`OpenClickyModelCatalog.swift:149-151`), so any non-gpt-5.x ID maps silently to gpt-5.5 and never reaches `turn/start`. The surviving true point: **Agent Mode is Codex-only; the `.openAI` provider tag on those options is inert; there is no Claude agentic Agent-Mode path at all.**

**Divergence 4 — money-rule enforcement is consistent where it bills.** SDK/app-server-first is correct for voice (`analyzeClaudeResponse` 15341; `analyzeOpenAIOrCodexVoiceResponse` 15402). After verification, the two previously-flagged "violations" are reclassified:
- *`ElementLocationDetector` proactive pointing* (`CompanionManager.swift:15797-15805`): direct REST, but **no SDK alternative exists** for the `computer_20251124` beta tool (D1). Not a violation.
- *`CodexAgentSession.fastFriendlyTitle`* (`CodexAgentSession.swift:1659-1726`): **on-device FoundationModels first** (free, local, live on this macOS 26 host), then a 64-token-capped ClaudeAPI/OpenAIAPI REST fallback. Not "bills per token every use." It *is* a genuine SDK-skip when it does fall to Anthropic, but the cost is negligible — keep the direction (prefer SDK/local in any automation loop), drop the severity.
- The one real defect is the **`voiceResponseExecutionFields` logging inversion** (3520-3539): logs `api_key_primary` while the runtime is SDK-first. Mis-reports, does not mis-bill — fix for observability.

**Unify plan (proposed abstraction).**
1. `struct AgentRequest { instruction; origin (.voiceRealtimeToolCall(toolName:) / .voiceTranscriptCascade / .hud / .programmatic); screenContext; preferredProvider; interruptVoiceResponse; requestID }` — `origin` carries the realtime `toolName` so re-classification is skipped when the model already classified.
2. `protocol AgentExecutor { var capabilities; func start(_:) async -> AgentHandle; func cancel(_:) async }` with a unified progress stream mapping `CodexAgentProgressStage` / SDK delta-result / daemon token-done into one enum. `CodexAgentExecutor` wraps today's `CodexAgentSession` (no day-one behavior change); future `ClaudeAgentSDKExecutor` / `OpenClickyDaemonExecutor` implement the same protocol.
3. `AgentRouter`: `routeFinalVoiceTranscriptActionIfNeeded` becomes a pure classifier producing an `AgentRequest` (or nil); one dispatch site picks the executor by capability/provider; the realtime handler builds `AgentRequest` with `origin=.voiceRealtimeToolCall(toolName)` and honors it. This collapses 6+ switch sites into one capability registry and makes the money rule a single shared policy in `executor.start`.

### 2.4 System Prompts & Skills — Portability + Streamline

**VERDICT: Skill content is provider-neutral markdown; the loading machinery is hard-bound to Codex. There is real, removable duplication (a 17KB dead-cargo instructions file + 7 clicky-/openclicky- skill pairs), and the single biggest structural target is the 17,194-line `CompanionManager.swift`.**

**Provider-neutrality of prompts.**
- *Voice prompt:* provider-neutral. `companionVoiceResponseSystemPrompt` (`CompanionManager.swift:14096-14155`) and `companionRealtimeVoiceSystemPrompt` (14157-14189) contain zero provider names — persona + pointing-syntax only. (The assembled prompt actually sent via `currentVoiceResponseSystemPrompt()` injects a `runtimeStorageContextForVoicePrompt()` path list that names a literal "codex home" at 14197 — a path helper, not the persona.)
- *Agent instructions:* mixed. `AGENTS.md:3` is provider-neutral ("You are running as an OpenClicky background agent."). But `OpenClickyModelInstructions.md:1` opens "You are OpenClicky's Codex Agent Mode." and references "Codex home" 5+ times (lines 11, 15, 16, 17). The live instructions file is Codex-framed in persona even though its content (memory discipline, archive-first, gog routing) is portable.

**Skill loading is Codex-bound by construction.** `config.toml` emits two `[[skills.config]]` blocks with `model_instructions_file` and `bundled_skills_dir` (`ClickyCodexConfigTemplate.swift:119-129`) — Codex CLI concepts. `ClaudeAgentSDKAPI.swift` has zero references to config.toml, skills, `bundled_skills_dir`, or `model_instructions`. The corpus markdown would work against any model that can read instructions, but the machinery that *loads* it exists only for Codex. A future Claude-Agent-SDK lane would need a fresh bootstrap mirroring `CodexHomeManager`. (D1 confirms the Claude Agent SDK reads Claude Code filesystem config — but through its own mechanism, not Codex's config.toml; D3 confirms Codex loads MCP/skills from `~/.codex/config.toml`. Two genuinely different config surfaces; a shared corpus needs a thin per-runtime bootstrap, not a shared loader.)

**Duplication — `ClickyModelInstructions.md` is dead cargo.** The build phase (`project.pbxproj:409`) ditto-copies it (17,380 bytes) into the bundle, but `modelInstructionsFileName` is hardcoded to `OpenClickyModelInstructions.md` (`CodexHomeManager.swift:20`, `ClickyCodexConfigTemplate.swift:22`). No code path assigns the Clicky file — it ships but is never loaded. **Correction:** the live `OpenClickyModelInstructions.md` does **not** route to `clicky-*` skills — grepping it for `clicky-`/`openclicky-` returns zero matches. The stale `clicky-*` routing table (lines 33-41) lives only in the *dead* `ClickyModelInstructions.md`, which is therefore the sole surviving reference to the `clicky-*` family in any instructions file — strengthening the case to delete it.

**Duplication — clicky-/openclicky- skill pairs.** Seven pairs ship simultaneously into the same `bundled_skills_dir` (artifacts, build-preview, creative-studio, dev-setup-doctor, email-assistant, repo-operator, research-report). `clicky-google-workspace` has no openclicky- twin (superseded by `google-workspace-gogcli`). Since the live instructions no longer name `clicky-*`, the clicky-* folders are matched only by natural-language name overlap — redundant with their openclicky- twins (which carry the `_shared` compatibility-policy header).

**Voice vs agent skill sharing (nuanced).** They do **not** share a skill set at the loading layer. Voice-lane skill awareness = `OpenClickyAppSkillContext.swift`, a hardcoded **10-app** table (Premiere, Final Cut, Terminal, Xcode, Figma, Excel, VSCode, Shopify, Notion, Blender — `grep -c "OpenClickyAppSkillContext("` returns 10) compiled into the binary, plus the bundled-skills directory *path* injected as a prompt string — never SKILL.md content. Background agents get the full corpus via config.toml. The voice lane reaches the corpus only by delegating to the Codex agent (`openclicky_start_background_agent` / `startVoiceAgentTaskPlan`). So skills are shared by delegation, not loading. A new bundled skill gains zero voice-lane effect until `OpenClickyAppSkillContext.swift` is hand-edited.

**Streamline plan.**
- *(a)* Delete `ClickyModelInstructions.md` from build + repo (dead, sole clicky-* routing survivor).
- *(b)* Collapse the 7 clicky-/openclicky- pairs to the openclicky- set (verify Codex resolves by directory name vs YAML `name:` first; also fix `OpenClickyAgentStore.swift:195` which registers `clicky-repo-operator`).
- *(c)* Extract the routing cascade out of `CompanionManager` into a `VoiceRouter` returning a typed `RouteDecision` — the precondition for every other cleanup and the path to unit-testing routing.
- *(d)* Introduce a `SkillCorpus` abstraction (parse each `SKILL.md` YAML frontmatter) so voice and any future agent lane resolve skills from one source rather than a hardcoded Swift table + Codex-only config.toml.
- *(e)* Make the agent instruction persona provider-neutral (`OpenClickyModelInstructions.md` lines 1, 11, 15, 16, 17: "Codex Agent Mode"/"Codex home" → runtime-neutral) so the same file can drive a future Claude/OpenAI lane.

### 2.5 Voice-Agent Capability Gaps

**VERDICT: The voice lane is more capable than "gaps" implies — app launch, folder/web open, volume, type, key, click, Spotify, reminders, messages search all work locally. The one genuinely missing capability is inline web search. Secondary: uncached clicks use weaker vision while the accurate beta tool only feeds the overlay and the cached-click path; bare-utterance launch is whitelist-gated.**

**What the voice lane already does directly** (`handleDirectComputerUseRequest`, `CompanionManager.swift:4424, 4710-5062`): folder open, web open, composite app+action, Spotify, system volume, local app launch, reminder add/count, messages search, native click/type/key. Live-partial pre-execution (4444-4582) even fires app/folder/key before finalization.

**Gap 1 — inline web search (biggest).** `containsFreshResearchRequest` (11183-11186) matches latest/live/price/news/weather/research/look up/search the web/google/browse and routes them exclusively to the agent path. The voice-response inference has no search tool: the Claude Agent SDK bridge is a stateless tool-less relay (R5). So "what's the weather" spins up a Codex agent. **D1 confirms Claude has a server-side `web_search` tool (4.x, streaming, citations); D3 confirms Codex first-party web search.** Fillable, not a model limit. For `.anthropic`, route through the SDK-first path (money rule) — which requires the SDK bridge to expose the server `web_search` tool; for `.codex`/`.openAI`, use Codex first-party search.

**Gap 2 — click accuracy.** The accurate `computer_20251124` beta tool (`ElementLocationDetector.swift`) drives the cursor overlay (`CompanionManager.swift:15799`) and — **correction** — feeds a *real* click only via the cached `prefersLastPointedElement` path (within 120s, 6389-6402). The uncached/primary click path uses the weaker `[POINT:x,y]` vision tag (`parsePointingCoordinates`, 6421/6428). Routing the uncached path through the beta detector for the `.anthropic` branch is a contained accuracy win for the most error-prone action. Keep the framing as "pixel-counting training is more accurate than standard vision" (`ElementLocationDetector.swift:14-16` comment); the "1:1 pixel coordinate" benefit is Opus-4.8/4.7-only per D1 and is **not** realized on what the app runs (catalog Opus = `claude-opus-4-6`; detector default = `claude-sonnet-4-6`, downsampled ≤1366px with scaling).

**Gap 3 — app-launch breadth.** Bare-utterance launch is whitelist-gated (`isKnownBareLocalApplicationName`); only verb-prefixed launches use the general regex. Apps not in the alias table and not verb-prefixed silently fall through.

**Per-provider asymmetry (drives priority).** OpenAI realtime — tools wired, event model exists, supervisor-delegation is OpenAI's own recommended pattern (D2); lowest lift (just consume the decision). Deepgram — no tools wired (R4); D3 confirms Deepgram supports client-side function calling; medium lift to parity. Non-realtime push-to-talk — no model in the routing loop at all; largest lift (needs mid-stream tool handling the realtime path already has).

**Distribution caveat (D1, not blocking for personal use):** the SDK-first "reuse paid Claude Code sign-in" rationale holds for personal/local use; Anthropic disallows third-party products offering claude.ai login to end users, and from 2026-06-15 subscription Agent SDK usage draws a separate monthly credit. Relevant only if web_search-via-SDK is later distributed.

### 2.6 The IT-vs-Agent-vs-Both Decision Policy

**VERDICT: Keep the regex cascade as the sub-100ms deterministic fast-path for unambiguous local actions, but make the model's tool choice authoritative for the ambiguous IT-vs-agent decision — the realtime model already emits that decision and OpenClicky discards it. The decision mechanism should be tiered, not a single global mechanism. Recommend: consume the existing tool choice (do-now), then add a model do-now-then-dispatch tool reusing the existing hybrid mechanism (then-fix), attaching web_search to the same inference.**

**The wasted routing signal (central finding).** Realtime sessions carry two tools (`ElevenLabsTTSClient.swift:2062-2095`): `openclicky_use_computer` ("open apps, app+action, typing, key, clicking") and `openclicky_start_background_agent` ("code, files, research, settings, logs, builds, refactors, long-running"). The model is explicitly prompted to make the IT-vs-agent decision and emits it as its tool choice. `firstFunctionCall` extracts (name, arguments) and calls `routeRealtimeToolCall` (`CompanionManager.swift:4060-4076`), which logs `toolName` (4067) then calls `routeCompletedRealtimeVoiceTranscriptIfNeeded(transcript)` (4075) — passing only the transcript. The full hardcoded cascade re-derives the decision. The LLM routing surface is already wired and (on realtime) already paid for; OpenClicky ignores its output.

**Is the cascade the right mechanism?** R3 establishes ~100% hardcoded routing: ~13 ordered guard-and-return checks, zero LLM classification (`shouldEscalateVoiceResponseToAgent` at 11019 is the one model-in-loop function and is dead code). Brittleness: `permissiveAgentInstruction` fires on any sentence containing "agent"; `containsAgentWorkAction` (~40 verbs incl. check/find/read) × `containsDurableWorkTarget` (~50 nouns incl. file/code/model) covers most sentences; the hybrid trigger needs **both** a foreground cue AND a literal background cue-word, so "explain why this is broken and fix it" does **not** trigger hybrid. But the cascade earns its keep for deterministic sub-100ms routing of clear local actions ("open Safari", "volume up") with no model round-trip or per-token cost. The verdict is tiered, not binary.

**Recommended tiered design.**
- *Layer 1 (keep):* regex fast-path for unambiguous direct local actions and cancellation/status/selection meta-commands.
- *Layer 2 (change):* make the model's tool choice authoritative for IT-vs-agent-vs-both. Three tools on the inference *already happening*: `openclicky_use_computer` [exists → `handleDirectComputerUseRequest`]; `openclicky_start_background_agent` [exists → agent launch]; **new** `openclicky_do_now_then_dispatch(quick_action, agent_instruction)` hybrid tool replacing the regex hybrid trigger, built on the existing deliberate false-return mechanism (`startHybridAgentTaskIfNeeded` 4415-4417 lets the foreground answer fire concurrently with the background agent) — no new plumbing. Crucially attach these to the **same call already running** — do *not* add a separate upfront classifier inference (extra round-trip + per-token cost). On realtime this is free: the fix is to consume `toolName` in `routeRealtimeToolCall`.
- *Layer 3 (extend):* add `web_search` inline on the voice-response call (Claude web_search via SDK-first per money rule, or Codex first-party for `.codex`/`.openAI`) — so factual questions get one-shot spoken answers. Same mechanism fixes routing AND search.

The do-now-then-fix hybrid is therefore native: the foreground answer fires immediately (do-now) while the background agent is dispatched concurrently (then-fix), and the model — not the regex — decides when that split is warranted.

### 2.7 Learning Loop — Logs → Issues → Self-Modification

**VERDICT: The pieces all exist; orchestration and gating do not — this is wiring, not a rebuild. And it is *more* wiring-shaped than first thought: the previously-claimed repo-operator "blocker" is not a blocker, so no functional blocker stands between today and a safe gated loop.**

**Today.** Rich multi-lane JSONL logging (`OpenClickyMessageLogStore`): seven lanes, daily rotation, NSLock+writeQueue, key/token redaction, end-to-end request timing. A human-annotation bridge: the log viewer lets the user flag entries as review comments, written to `log-review-comments.jsonl` + `agent-review-comments.md`, which is injected into every agent task brief (`CodexAgentSession.swift:674-677, 767`) and the voice system prompt. Two bundled skills (`learn-from-openclicky-logs`, `optimize-openclicky-skills`) and a `specialist-builder` agent that can create agent definitions and SKILL.md files. A 30s automation scheduler (`OpenClickyAutomationStore`) with one system automation (App skill discovery, 6h interval) — **disabled by default** and scoped to Connect-tab install suggestions, not self-repair.

**What is missing.** No automatic closed loop: no route-miss logging (the cascade logs routes *taken*, never fall-through, so misroutes are invisible), no scheduled digest that turns logs into issues, no dev-mode gate for self-modification. HEARTBEAT files are empty scaffolding never written by a live agent.

**Correction (high — bug-is-intent):** the repo-operator "skill-ID mismatch" is **not a mismatch and not a blocker**. `OpenClickyAgentDefinition.swift:112-121` defines `legacySkillAliases` mapping `clicky-repo-operator` → `openclicky-repo-operator`; the resolver (156-169) builds `[requestedID] + aliases` and returns the first match. The `clicky-repo-operator/SKILL.md` directory exists, is non-empty (2461 bytes), and passes `safeSkillID`, so the enabled ID resolves on the **first** candidate — the alias is only a backstop. The clicky-/openclicky- duplication is intentional, runtime-handled legacy-rename maintenance debt, not a functional blocker. Reclassify from P1 blocker to a low-priority cleanup. (Note also: the previously-cited evidence `OpenClickyModelInstructions.md:33-41` does not contain repo-operator content — the actual locus is `specialist-builder/skills.json:8` + the alias map.)

**Correction (low):** `fastFriendlyTitle` is not REST-first. Order is on-device FoundationModels (free, local, macOS 26+) → ClaudeAPI REST `claude-haiku-4-5` capped at 64 output tokens → OpenAIAPI REST `gpt-5.4-mini`. On this host the primary path is free local inference. Keep the direction (avoid REST in an automation loop) but drop the "money-leak" severity.

**Target design + safety rails.**
- Add a `route_settled` log event recording matched/unmatched routing-check names, so mining can distinguish "correctly handled by voice" from "missed agent route."
- Add a scheduled read-only digest automation that invokes `learn-from-openclicky-logs`, producing memory updates — low-risk, no code edits.
- Add a **dev-mode-gated** repair dispatch that is branch-and-PR-only, no auto-merge (`devMode` flag + `openClickyRepoPath`). The specialist-builder *can* edit Swift source today via the `codex` / repo-operator skills (`OpenClickyAgentStore.swift:189-197`) — but only when explicitly user-directed. The safety rail is: gate any log-driven self-edit behind dev-mode + branch + PR, and keep self-edited skills canonical-openclicky-only, archive-first, in `LearnedSkills`.
- Suppress or local-/SDK-route `fastFriendlyTitle` for automation sessions so a loop never trickles REST cost.

## 4. Risk & Money-Rule Audit

**Money-rule status: clean where it bills.** The rule (SDK / app-server first, REST keys fallback only) is correctly implemented and enforced on the paths that actually bill per token:
- Voice `.anthropic`: `analyzeClaudeResponse` tries `claudeAgentSDKAPI` first, falls to `claudeAPI` HTTP only on SDK failure with a key present (`CompanionManager.swift:15341-15391`). Correct.
- Voice `.openAI`/`.codex`: `analyzeOpenAIOrCodexVoiceResponse` / `analyzeCodexVoiceResponse` try the local Codex app-server / OpenAI key in the right order (15393-15458). Correct.
- Computer-use pointing `.anthropic` (click execution): routes through `analyzeClaudeResponse` (SDK-first). Correct.
- Browser Workspace CUA: SDK-only, throws if SDK nil — no inadvertent REST (`CompanionManager.swift:17044-17060`). Correct.

**Reclassified — not violations (verification applied):**
- **`ElementLocationDetector` proactive pointing** (`CompanionManager.swift:15797-15805`): issues a `computer_20251124` beta call (`ElementLocationDetector.swift:185`, beta header `:166`). Per D1, computer/desktop control is **not** an Agent SDK built-in tool — there is no SDK transport for the beta header. Direct REST is architecturally necessary, not a defect. Real per-use cost; mitigate by **gating when proactive pointing fires**, not by forcing it through an SDK that cannot carry it.
- **`CodexAgentSession.fastFriendlyTitle`** (`CodexAgentSession.swift:1659-1726`): tries on-device FoundationModels first (free, local, live on this macOS 26 host), then a **64-token-capped** ClaudeAPI/OpenAIAPI REST fallback. It is a genuine SDK-skip *if* it falls to Anthropic, but the cost is negligible and it is not "per-token on every use." Keep the cleanup direction (avoid REST in any automation loop); drop the severity.

**The one real defect (observability, not billing):** `voiceResponseExecutionFields` (`CompanionManager.swift:3520-3539`) logs `authMode: anthropic_api_key_primary` / `executionMethod: ClaudeAPI.analyzeImageStreaming` whenever a key is present, while `analyzeClaudeResponse` actually runs SDK-first. Latency and backend attributed to the SDK path get mis-tagged as ClaudeAPI in dashboards. It **mis-reports, it does not mis-bill.**

**The `.codex` vs `.openAI` catalog question — resolved definitively.** This is an intentional cost-routing split, not a bug, on top of one inert tag:
- The **same gpt-5.x ID** carries `provider:.openAI` in `voiceResponseModels` and `codexActionsModels`, and `provider:.codex` in `computerUseModels` (`OpenClickyModelCatalog.swift:61-103`).
- In **voice response**, `.openAI` means "try the direct OpenAI key first, fall back to the Codex app-server" — key-first.
- In **computer-use pointing**, `.codex` means "skip the key path entirely, use the local Codex subprocess (`CodexPointDetector`)" — app-server-only. Pointing never tries the key path. This is deliberate (cheaper, no key required) and consistent with the money rule per path.
- In **`codexActionsModels` (Agent Mode)**, the `.openAI` tag is **inert**. `CodexAgentSession` always runs the bundled local Codex process (`CodexAgentSession.swift:736-742`) and coerces any non-gpt-5.x ID to gpt-5.5 via `codexActionsModel(withID:)` → `codexActionsModels[0]` (`OpenClickyModelCatalog.swift:149-151`). No `openAIAPI` call path exists in `CodexAgentSession`. The provider value does not affect billing or transport. **There is no Claude agentic Agent-Mode path at all** — the only way to influence Agent Mode is the gpt-5.x model choice.

**Other risks worth flagging (non-billing):**
- Realtime visual fallback always resolves to gpt-5.4/.openAI (`CompanionManager.swift:14441-14444`) even with only an Anthropic key — a key-availability footgun, not a money-rule break.
- `CodexPointDetector` and the voice `CodexVoiceSession` run with `danger-full-access` / `approvalPolicy: never` — broad filesystem scope for narrow tasks; a screenshot-prompt-injection surface.
- Hardcoded dev paths in `OpenClickyComputerUseRuntime.swift:138-141` and `ClaudeAgentSDKAPI.swift:700-705` are dev-convenience leftovers; production resolves correctly via bundle lookup, but they should not ship.
